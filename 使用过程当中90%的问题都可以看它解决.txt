## 📋 软件说明

NodeCollector 是一款用于管理和配置网络节点的工具，支持：
- 多协议节点解析与验证
- 本地配置文件导入与管理
- 节点信息可视化展示
- 配置导出（多种格式）

本软件仅用于合法的网络配置管理，请遵守当地法律法规。


## 🚀 快速开始

### 第一步：启动软件

1. 双击 `NodeCollector.exe`
2. 首次启动会看到黑色窗口（这是程序后台，请勿关闭）
3. 等待 3-5 秒，浏览器会自动打开
4. 如果没有自动打开，手动访问：`http://127.0.0.1:5000`

### 第二步：主界面介绍

打开后你会看到：
- **协议筛选**：选择需要管理的协议类型（VMess、VLESS、SS、Trojan、Hysteria2）
- **数据源**：选择节点配置的来源
- **国家/地区筛选**：按地区过滤节点
- **线程数**：并发处理数量（默认50）
- **最大节点数**：限制获取的节点总数（默认100）

### 第三步：导入或获取节点

#### 方式A：导入本地配置文件（推荐）
1. 点击「导入本地库」按钮
2. 选择你的 `.txt` 配置文件（每行一个节点URI）
3. 选择需要筛选的协议
4. 点击「上传」，等待处理完成

#### 方式B：从数据源获取
1. 勾选需要的协议（至少选一个）
2. 选择数据源（默认"全"表示所有源）
3. 设置线程数和最大节点数
4. 点击「开始获取」按钮
5. 等待进度条完成

### 第四步：导出配置

1. 获取完成后，选择导出格式：
   - `geekez`：通用格式
   - `v2rayn`：适配 V2rayN 软件
   - `txt`：纯文本格式
2. 选择要导出的节点来源（当前/本地/网络）
3. 选择地区筛选（可选）
4. 点击「导出」按钮
5. 文件会保存到 `nodes` 文件夹

### 第五步：其他功能

- **复制到剪贴板**：一键复制所有节点到剪贴板
- **打开文件夹**：快速打开 nodes 输出目录
- **清空文件夹**：清空 nodes 目录下的所有文件
- **清空历史**：清空已处理节点的历史记录


## ⚠️ 常见问题与修复

### 问题1：双击 EXE 后没反应

**现象**：双击后没有任何窗口出现

**原因**：
- Windows 防火墙阻止
- 杀毒软件误报隔离
- 端口 5000 被占用

**修复方法**：
1. 检查是否被杀毒软件隔离：
   - 打开杀毒软件隔离区
   - 将 NodeCollector.exe 恢复并添加信任

2. 手动以管理员身份运行：
   - 右键 NodeCollector.exe
   - 选择「以管理员身份运行」

3. 检查端口占用：
   - 按 `Win + R`，输入 `cmd`，回车
   - 输入：`netstat -ano | findstr "5000"`
   - 如果显示有进程占用，打开任务管理器结束该进程


### 问题2：浏览器无法打开或显示"无法访问此网站"

**现象**：程序启动了，但浏览器显示"无法访问此网站"

**原因**：
- Flask 服务未成功启动
- 端口 5000 被其他程序占用
- 防火墙阻止本地连接

**修复方法**：
1. 检查程序是否真的在运行：
   - 按 `Ctrl + Shift + Esc` 打开任务管理器
   - 查找 `NodeCollector.exe` 进程
   - 如果没有，重新双击启动

2. 手动访问：
   - 打开浏览器，地址栏输入：`http://127.0.0.1:5000`
   - 或试试：`http://localhost:5000`

3. 检查防火墙：
   - Windows 安全中心 → 防火墙和网络保护
   - 允许应用通过防火墙 → 找到 NodeCollector → 勾选专用和公用

4. 更换端口（高级用户）：
   - 如果 5000 端口一直被占用
   - 需要修改源码并重新打包（见文末）


### 问题3：导入文件后提示"未选择文件"或"请上传.txt文件"

**现象**：上传后显示错误提示

**原因**：
- 文件格式不对（不是 .txt）
- 文件编码不是 UTF-8
- 文件路径包含中文或特殊字符

**修复方法**：
1. 确保文件是 `.txt` 格式：
   - 右键文件 → 属性 → 确认扩展名是 .txt
   
2. 转换文件编码为 UTF-8：
   - 用记事本打开文件
   - 文件 → 另存为
   - 编码选择「UTF-8」
   - 保存后重新导入

3. 简化文件路径：
   - 把文件放到简单路径，如 `C:\nodes.txt`
   - 避免中文路径和特殊字符


### 问题4：获取节点时进度条不动或卡住

**现象**：点击获取后，进度条一直为0或卡住

**原因**：
- 数据源网络不可达（超时）
- 线程数设置过高导致程序卡死
- 选择的协议没有匹配的节点

**修复方法**：
1. 减少线程数：
   - 将线程数从 50 改为 10 或 20
   - 降低并发压力

2. 减少数据源：
   - 不要选"全"，手动选择 2-3 个数据源
   - 避免同时请求太多外部地址

3. 增加超时时间：
   - 默认超时是 10 秒
   - 如果网络慢，需要修改源码（见文末）

4. 检查日志：
   - 页面下方有日志区域
   - 查看是否有"获取失败"的红色日志
   - 根据错误信息调整


### 问题5：导出文件后找不到文件

**现象**：提示导出成功，但找不到文件

**原因**：
- 不知道输出目录位置
- nodes 文件夹未自动创建
- 权限问题无法写入

**修复方法**：
1. 找到输出目录：
   - 点击页面上的「打开文件夹」按钮
   - 或手动打开：`NodeCollector.exe` 同目录下的 `nodes` 文件夹

2. 手动创建文件夹：
   - 在 `NodeCollector.exe` 同目录创建 `nodes` 文件夹
   - 重新运行程序

3. 检查写入权限：
   - 确保 NodeCollector.exe 所在目录可写
   - 如果放在 C 盘根目录，可能需要管理员权限


### 问题6：Windows 防火墙弹窗提示

**现象**：首次启动弹出"是否允许访问"

**修复方法**：
1. 勾选「专用网络」和「公用网络」
2. 点击「允许访问」
3. 如果不小心点了「取消」：
   - 打开 Windows 安全中心
   - 防火墙和网络保护 → 允许应用通过防火墙
   - 找到 NodeCollector → 启用


### 问题7：杀毒软件报毒（误报）

**现象**：360、火绒、Windows Defender 报毒

**原因**：PyInstaller 打包的程序常被误报

**修复方法**：
1. 将 NodeCollector.exe 添加到白名单/信任区
2. 如果还是不行，可以：
   - 暂时关闭杀毒软件
   - 运行程序
   - 运行后再开启杀毒软件


### 问题8：程序闪退或崩溃

**现象**：启动后立即关闭，或运行中突然崩溃

**修复方法**：
1. 检查系统要求：
   - Windows 10 或更高版本
   - 已安装 Visual C++ Redistributable（通常系统自带）

2. 查看错误日志：
   - 在 `NodeCollector.exe` 同目录查找 `.log` 文件
   - 或打开命令提示符，拖入 EXE 运行，查看错误信息

3. 重新下载：
   - 可能文件下载不完整
   - 重新获取完整的 EXE 文件

### 问题9：导出文件无法导入到其他软件（v2rayN、Clash）

**现象**：导出的配置文件在其他软件中无法识别或报错

**原因**：
- 导出格式选择错误
- 节点 URI 格式不完整
- 目标软件版本不兼容

**修复方法**：
1. 选择正确的导出格式：
   - v2rayN 软件 → 选择 `v2rayn` 格式
   - Clash 软件 → 选择 `geekez` 格式（通用）
   - 不确定 → 选择 `txt` 格式（纯节点列表）

2. 检查节点有效性：
   - 在 NodeCollector 中查看节点数量
   - 如果节点数为 0，先获取/导入节点

3. 手动验证节点格式：
   - 用记事本打开导出的 `.txt` 文件
   - 确认每行都是 `协议://` 开头的完整 URI
   - 如果格式异常，重新导出

### 问题10：所有节点都无法连接或已失效

**现象**：导出的节点在其他软件中全部无法使用

**原因**：
- 数据源的节点已过期（免费节点通常寿命短）
- 网络环境限制（某些地区无法访问）
- 节点本身质量差

**修复方法**：
1. 重新获取最新节点：
   - 点击「开始获取」刷新数据源
   - 勾选更多数据源增加成功率

2. 使用"导入本地库"：
   - 如果有其他来源的节点文件
   - 通过「导入本地库」功能导入

3. 检查网络环境：
   - 尝试访问 `https://www.google.com` 测试连通性
   - 某些地区可能需要特殊网络环境

4. 降低期望：
   - 免费节点通常存活时间短（几小时到几天）
   - 建议每天刷新获取新节点

### 问题11：程序占用内存过高或 CPU 使用率高

**现象**：程序运行后内存占用持续增长，或 CPU 占用 100%

**原因**：
- 采集的节点数量过多（超过 1000 个）
- 线程数设置过高（100+）
- IP 历史记录累积过多（未清理）

**修复方法**：
1. 限制节点数量：
   - 设置"最大节点数"为 200-500
   - 避免设置过高（如 1000+）

2. 降低线程数：
   - 将线程数改为 20-30
   - 减少并发压力

3. 清理历史记录：
   - 点击「清空历史」按钮
   - 释放内存中的历史数据

4. 重启程序：
   - 关闭程序后重新启动
   - 内存会重新初始化

### 问题12：无法访问 GitHub（数据源全部失败）

**现象**：点击获取后所有数据源都显示"获取失败"

**原因**：
- 网络无法访问 GitHub（raw.githubusercontent.com）
- gh-proxy.com 代理服务不可用
- DNS 解析失败

**修复方法**：
1. 测试 GitHub 连通性：
   - 浏览器访问 `https://raw.githubusercontent.com`
   - 如果无法访问，说明网络环境受限

2. 更换网络环境：
   - 尝试切换 WiFi / 热点
   - 某些公共网络会屏蔽 GitHub

3. 手动下载节点文件：
   - 从浏览器访问数据源链接
   - 保存为 `.txt` 文件
   - 使用「导入本地库」功能导入

4. 使用备用数据源：
   - config.json 中配置的站点可能更稳定
   - 尝试使用 GUI 版（运行爬虫程序.py）从配置站点采集

### 问题13：导入本地文件后节点数量不对

**现象**：导入一个有 100 个节点的文件，但程序只显示 50 个

**原因**：
- 文件中有重复节点（被自动去重）
- 部分行格式不正确（不是有效的节点 URI）
- 选择了协议筛选，过滤掉了其他协议

**修复方法**：
1. 检查文件格式：
   - 每行必须是完整的节点 URI（以 `协议://` 开头）
   - 不能有空行或注释行

2. 查看日志：
   - 导入后查看页面下方日志
   - 会显示"跳过重复节点 X 个"

3. 不过滤协议：
   - 导入时勾选所有协议（VMess/VLESS/SS/Trojan/Hysteria2）
   - 或单独导入每种协议

4. 手动检查文件：
   - 用记事本打开文件
   - 搜索 `vmess://`、`ss://` 等关键字
   - 确认每种协议的数量

### 问题14：页面显示乱码或中文显示为方块

**现象**：页面上的中文显示为方块、问号或乱码

**原因**：
- 浏览器编码设置错误
- 系统缺少中文字体
- 程序输出的编码不正确

**修复方法**：
1. 设置浏览器编码：
   - Chrome/Edge：右键页面 → 编码 → Unicode (UTF-8)
   - 或地址栏输入 `chrome://settings/languages` 设置语言

2. 检查系统字体：
   - 确保系统已安装中文字体（如微软雅黑）
   - Windows 10+ 通常自带，无需额外安装

3. 重启程序：
   - 关闭 NodeCollector.exe
   - 重新双击启动

### 问题15：同时运行多个程序实例导致冲突

**现象**：双击多次后运行了多个 NodeCollector.exe，导致端口冲突

**原因**：
- 重复启动程序，没有关闭之前的实例
- 端口 5000 被第一个实例占用

**修复方法**：
1. 检查运行中的实例：
   - 按 `Ctrl + Shift + Esc` 打开任务管理器
   - 查找所有 `NodeCollector.exe` 进程
   - 全部结束任务

2. 只运行一个实例：
   - 启动后等待浏览器打开
   - 不要重复双击 EXE

3. 如果端口冲突：
   - 按 `Win + R`，输入 `cmd`
   - 执行 `taskkill /f /im NodeCollector.exe`
   - 重新启动程序

### 问题16：nodes 目录被误删后程序无法运行

**现象**：手动删除了 nodes 目录，程序启动报错或无法导出

**修复方法**：
1. 重新创建目录：
   - 在 `NodeCollector.exe` 同目录
   - 右键 → 新建 → 文件夹
   - 命名为 `nodes`

2. 重启程序：
   - 目录会自动初始化
   - `ip_history.json` 会重新生成

3. 如果仍有问题：
   - 完全关闭程序
   - 删除整个 `nodes` 目录
   - 重新启动，让程序自动创建

### 问题17：导出的 v2rayN 格式节点无法识别

**现象**：导出为 v2rayN 格式后，在 v2rayN 软件中导入失败

**原因**：
- v2rayN 版本过旧，不支持新格式
- 节点 URI 编码有问题
- 导出时选择了错误的协议筛选

**修复方法**：
1. 更新 v2rayN 软件：
   - 下载最新版 v2rayN
   - 旧版可能不兼容新的节点格式

2. 尝试 TXT 格式：
   - 在 NodeCollector 中导出为 `txt` 格式
   - 在 v2rayN 中选择"从剪贴板导入"
   - 手动粘贴节点 URI

3. 检查节点内容：
   - 用记事本打开导出的文件
   - 确认是有效的 JSON 格式（v2rayN 格式）
   - 或纯节点 URI 列表（TXT 格式）

### 问题18：程序运行正常但采集不到任何节点

**现象**：进度条走完，显示"获取完成"，但节点数为 0

**原因**：
- 所有数据源都失效或不可达
- 选择了不存在的协议筛选
- 网络完全无法访问外部资源

**修复方法**：
1. 取消协议筛选：
   - 勾选所有协议（不要只选一个）
   - 某些数据源可能只包含特定协议

2. 检查数据源状态：
   - 查看页面下方日志
   - 如果全部显示"获取失败"，说明网络问题

3. 使用本地导入：
   - 暂时放弃网络采集
   - 从其他途径获取节点文件
   - 使用「导入本地库」功能

4. 尝试 GUI 版：
   - 运行 `运行爬虫程序.py`
   - 使用 config.json 中配置的备用数据源

## ⚡ 性能优化建议

### 1. 线程数设置（关键优化）

线程数直接影响采集速度和稳定性：

| 线程数 | 适用场景 | 优点 | 缺点 |
|--------|----------|------|------|
| **10-20** | 网络较慢、不稳定 | 稳定、不易卡死 | 速度较慢 |
| **30-50** | 常规使用（推荐） | 速度适中、稳定性好 | 无明显缺点 |
| **50-100** | 高速网络、大量节点 | 速度快 | 可能触发反爬 |
| **100+** | 不推荐 | - | 容易卡死、被封 |

**最佳实践**：
- 首次使用：设置为 **20** 线程测试稳定性
- 日常使用：保持 **50** 线程（默认值）
- 网络差时：降低到 **10-15** 线程
- 数据采集：可临时提升到 **80** 线程（需监控日志）

### 2. 超时时间优化

默认超时设置：
- 连接超时：5 秒
- 读取超时：10 秒

**网络环境对应调整**：

```python
# 快速网络（光纤、5G）
timeout=(3, 8)   # 连接3秒，读取8秒

# 普通网络（4G、宽带）
timeout=(5, 10)  # 默认设置

# 慢速网络（公共WiFi、弱信号）
timeout=(10, 20)  # 连接10秒，读取20秒
```

**修改方法**：
1. 打开 `网页版NodeCollector.py`
2. 找到 `fetch()` 函数中的 `timeout` 参数
3. 按上表调整数值
4. 重新打包 EXE（见高级设置）

### 3. 数据源选择策略

**优先级排序**（按稳定性和速度）：

1. **FastNodes** 系列（最快、最稳定）
   - FastNodes All（全部节点）
   - FastNodes US（美国节点）
   - 按协议分类（VMess/VLESS/SS/Trojan/H2）

2. **NodesFree**（稳定但较慢）
   - 适合作为备份源

3. **EbraSha** 系列（节点质量高）
   - 但可能包含无效节点

4. **其他源**（作为补充）
   - ClashNode、V2RayClash（更新较慢）

**推荐组合**：
- **快速采集**：勾选 3-5 个 FastNodes 源
- **高质量**：FastNodes + EbraSha 组合
- **全面采集**：勾选"全"（但耗时较长）

### 4. 内存与存储优化

**避免内存溢出**：
- 设置合理的"最大节点数"（建议 100-500）
- 定期点击"清空历史"释放内存
- 避免一次性导入超大本地文件（>10MB）

**磁盘空间管理**：
- `nodes/` 目录会持续增长
- 定期手动清理旧的导出文件
- `ip_history.json` 会记录所有历史（可手动删除重置）

**清理命令**：
```bash
# 清理导出文件（保留历史记录）
del /q "D:\VPN\NodeCollector\nodes\*.txt"

# 完全重置（删除所有数据）
del /q "D:\VPN\NodeCollector\nodes\*.*"
```

### 5. 导出优化

**不同格式的性能差异**：

| 格式 | 导出速度 | 文件大小 | 适用场景 |
|------|----------|----------|----------|
| **TXT** | 最快 | 最小 | 快速备份、分享 |
| **GeekEZ** | 快 | 小 | 导入 GeekEZ 工具 |
| **v2rayN** | 较慢 | 较大 | 导入 v2rayN 软件 |

**批量导出建议**：
1. 先导出 TXT 格式测试
2. 确认节点有效后再导出 v2rayN 格式
3. 避免同时导出多个格式（逐个导出更稳定）

### 6. 并发采集最佳实践

**场景1：首次使用**
```
线程数：20
最大节点数：100
数据源：3-5 个 FastNodes
超时：默认
→ 目标：测试稳定性，避免卡死
```

**场景2：日常刷新**
```
线程数：50（默认）
最大节点数：200
数据源：勾选"全"
超时：默认
→ 目标：平衡速度与稳定性
```

**场景3：低速网络**
```
线程数：10-15
最大节点数：50
数据源：2-3 个 FastNodes
超时：(10, 20)
→ 目标：避免超时，提高成功率
```

**场景4：高速采集**
```
线程数：80-100
最大节点数：500+
数据源：勾选"全"
超时：(3, 8)
→ 目标：快速获取大量节点（需监控日志）
```

### 7. 浏览器性能

**减少浏览器负担**：
- 不要同时打开多个标签页采集
- 采集时避免播放视频、下载文件
- 定期刷新页面（F5）清理内存

**页面卡顿解决**：
1. 点击"清空历史"按钮
2. 关闭浏览器，重启程序
3. 减少显示的节点数量（设置最大节点数）

### 8. 日志与监控

**关注日志中的警告**：
- `获取失败`：数据源不可达，减少该源使用
- `超时`：网络慢，增加超时时间或减少线程
- `已存在`：节点重复，正常现象（去重生效）

**性能监控技巧**：
- 观察进度条速度（正常：每秒 2-5 个节点）
- 查看最终成功率（目标：>80%）
- 记录不同设置下的采集时间，找到最优配置

### B. API 接口说明

网页版 Flask 提供以下 API 接口（供高级用户调用）：

#### 基础接口

| 接口 | 方法 | 功能 | 参数 |
|------|------|------|------|

| `/api/progress` | GET | 获取进度 | 返回：progress（当前进度）、total（总数）、running（是否运行中） |

#### 数据采集接口

| 接口 | 方法 | 功能 | 参数 | 返回 |
|------|------|------|------|------|
| `/api/scrape` | POST | 开始采集节点 | `protocols`: 协议数组<br>`threads`: 线程数<br>`max_nodes`: 最大节点数<br>`source_count`: 数据源数量<br>`country`: 国家筛选 | `started`: 是否成功启动<br>`message`: 提示信息 |

| 接口 | 方法 | 功能 | 参数 | 返回 |
|------|------|------|------|------|
| `/api/import_local` | POST | 导入本地文件 | `file`: 文件对象<br>`protocols`: 协议列表（表单） | `count`: 导入数量<br>`protocols`: 各协议数量 |
| `/api/load_library` | POST | 加载离线库 | `protocols`: 协议列表（JSON）<br>`max_nodes`: 最大节点数 | `count`: 加载数量<br>`skipped`: 跳过数量 |
| `/api/export` | POST | 导出节点 | `country`: 国家<br>`format`: 格式（geekez/v2rayn/txt）<br>`protocols`: 协议列表<br>`source`: 来源（current/local/scraped） | `count`: 导出数量<br>`filename`: 文件名 |
| `/api/copy` | POST | 复制到剪贴板 | 无 | `success`: 是否成功<br>`count`: 节点数量 |

#### 文件管理接口

| 接口 | 方法 | 功能 | 参数 | 返回 |

#### API 调用示例

**使用 curl 测试：**

```bash
# 1. 获取日志
curl http://127.0.0.1:5000/api/logs

# 2. 获取进度
curl http://127.0.0.1:5000/api/progress

# 3. 开始采集（JSON格式）
curl -X POST http://127.0.0.1:5000/api/scrape \
  -H "Content-Type: application/json" \
  -d "{\"protocols\": [\"VMess\", \"VLESS\"], \"threads\": 20, \"max_nodes\": 100, \"source_count\": \"全\", \"country\": \"全部\"}"

# 4. 导出节点
curl -X POST http://127.0.0.1:5000/api/export \
  -H "Content-Type: application/json" \
  -d "{\"format\": \"txt\", \"source\": \"current\", \"country\": \"全部\", \"protocols\": [\"VMess\"]}"

# 5. 清空历史
curl -X POST http://127.0.0.1:5000/api/clear_history

# 6. 关闭服务
curl -X POST http://127.0.0.1:5000/api/shutdown
```

### C. 配置文件详解

#### config.json 结构

```json
{
  "数据源名称": {
    "start_url": "起始页面URL",
    "up_date": "更新日期 (YYYY-MM-DD)",
    "selector": "CSS选择器（提取链接）",
    "pattern": "正则表达式（匹配节点链接）",
    "password": "页面密码（可选）",
    "script": "自动填充密码的JS脚本（可选）",
    "yt_idx": "YouTube视频索引（可选）"
  }
}
```

#### 字段说明

| 字段 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `start_url` | 字符串 | 是 | 爬虫起始页面 | `"https://www.example.com/"` |
| `up_date` | 字符串 | 是 | 数据源更新日期 | `"2026-04-18"` |
| `selector` | 字符串 | 是 | CSS选择器，用于提取页面中的链接 | `"a[class='text-reset']"` |
| `pattern` | 字符串 | 是 | 正则表达式，匹配节点文件链接 | `"https?://example\.com/[^<>\\n]+"` |
| `password` | 字符串 | 否 | 页面访问密码（部分站点需要） | `"1122"` |
| `script` | 字符串 | 否 | 自动填充密码的JavaScript代码 | `"var input = document.querySelector('.cl-input');..."` |
| `yt_idx` | 整数 | 否 | YouTube视频索引（用于获取密码） | `0` |

#### 新增数据源示例

```json
{
  "mysource": {
    "start_url": "https://www.mysite.com/",
    "up_date": "2026-05-02",
    "selector": "a[href*='node']",
    "pattern": "https?://mysite\\.com/uploads/[^<>\\n]+"
  }
}
```

**添加步骤：**
1. 打开 `config.json`
2. 在最后一个数据源后添加逗号
3. 粘贴上述配置（修改实际值）
4. 保存文件
5. 重新运行 `运行爬虫程序.py`（GUI版）

### C2. 数据源管理（新增/删除/配置）

#### 数据源类型对比

| 类型 | 位置 | 使用方法 | 优点 | 缺点 |
|------|------|----------|------|------|
| **config.json** | NodeScrapy 爬虫源 | GUI版（运行爬虫程序.py） | 自动化、稳定 | 需要配置selector/pattern |
| **网页版内置** | `网页版NodeCollector.py` 的 `SOURCES` 变量 | 网页版直接使用 | 无需配置、即用 | 需修改源码重新打包 |
| **本地文件** | `nodes/离线IP库.txt` | 导入本地库功能 | 最灵活、可控 | 需手动维护文件 |

#### 新增数据源（网页版）

**步骤：**

2. 找到第 51-70 行的 `SOURCES = [` 列表

4. 保存文件
5. 重新打包 EXE（见下文"重新打包EXE"）

**注意事项：**
- 数据源必须是 `.txt` 文件，每行一个节点 URI
- 建议使用 `gh-proxy.com` 代理 GitHub 资源（如：`https://gh-proxy.com/https://raw.githubusercontent.com/...`）

GUI版使用 `config.json` 配置，无需重新打包：

#### 如何获取 selector 和 pattern

**selector（CSS选择器）：**
- 打开目标网页（如 `https://www.example.com/`）
- 按 `F12` 打开开发者工具
- 点击左上角"元素选择器"图标
- 点击页面上的节点链接
- 查看高亮的 HTML 代码，提取 CSS 选择器

**示例：**
```html
<!-- 页面HTML -->
<a class="node-link" href="/download/nodes.txt">下载节点</a>

<!-- selector 就是： -->
"selector": "a.node-link"
```

**pattern（正则表达式）：**
- 匹配节点文件的完整URL
- 从 selector 找到的链接中提取

**示例：**
```python
# 如果链接是：https://example.com/uploads/2026/05/nodes.txt
# pattern 就是：
"pattern": "https?://example\\.com/uploads/[^<>\\n]+"
```

### D. 协议格式说明

#### 支持的协议

| 协议 | 前缀 | 格式示例 | 特点 |
|------|------|----------|------|
| **VMess** | `vmess://` | `vmess://base64encodedjson` | 需要Base64解码获取配置 |
| **VLESS** | `vless://` | `vless://uuid@ip:port?encryption=none&...` | 轻量级，无加密 |
| **Shadowsocks** | `ss://` | `ss://base64encoded@ip:port` 或 `ss://base64encodedstring` | 传统协议，兼容性好 |
| **Trojan** | `trojan://` | `trojan://password@ip:port?security=tls&...` | TLS加密，安全性高 |
| **Hysteria2** | `hysteria2://` | `hysteria2://auth@ip:port?insecure=0&...` | 基于QUIC，速度快 |

#### URI 结构通用格式

```
协议://[认证信息]@[IP地址]:[端口]?[参数]#[备注]
```

**示例解析：**

1. **VMess**:
   ```
   vmess://eyJhZGRyZXNzIjoiMS4yLjMuNCIsInBvcnQiOiI0NDMiLCJ0eXBlIjoidGVscyIsImlkIjoiMTIzNDU2Nzg5MDEiLCJuZXQiOiJ3cyIsInBhdGgiOiIvcGF0aCIsInRscyI6InRscyJ9
   ```
   - 解码后为JSON配置：`{"add":"1.2.3.4","port":"443",...}`

2. **VLESS**:
   ```
   vless://12345678-1234-1234-1234-123456789012@1.2.3.4:443?encryption=none&security=tls&sni=example.com&type=ws#MyNode
   ```
   - UUID: `12345678-1234-1234-1234-123456789012`
   - IP: `1.2.3.4`
   - 端口: `443`
   - 备注: `MyNode`（#号后）

3. **Trojan**:
   ```
   trojan://password123@1.2.3.4:443?security=tls&sni=example.com&type=tcp#MyTrojan
   ```
   - 密码: `password123`
   - IP: `1.2.3.4`

#### IP 提取正则

程序使用以下正则提取IP和端口：

| 正则 | 用途 |
|------|------|
| `_RE_AT_IP_PORT` | 匹配 `@IP:端口` 格式（VLESS/Trojan/SS/Hysteria2） |
| `_RE_URI_HASH` | 分割URI和备注（`#`号后） |
| `_RE_PROTOCOL_START` | 匹配协议开头（`vmess://`, `vless://`等） |
| `_RE_IP_PORT_END` | 匹配末尾的 `IP:端口`（纯IP格式） |

### E. 数据文件格式

#### nodes.json（当前节点数据）

```json
[
  "vmess://...",
  "vless://...",
  "ss://...",
  "trojan://...",
  "hysteria2://..."
]
```

- 每行一个完整的节点URI
- 自动去除 `#` 后的备注后存储

#### ip_history.json（IP历史去重）

```json
{
  "hash1": true,
  "hash2": true,
  "...(更多MD5哈希)": true
}
```

- Key: 节点URI的MD5哈希值（`hashlib.md5(node_uri.encode('utf-8')).hexdigest()`）
- Value: 始终为 `true`
- 用途: 避免重复采集相同节点

#### 导出文件命名规则

```
国家代码_协议1_协议2.txt
```

**示例：**
- `US_VMess_VLESS.txt` - 美国的VMess和VLESS节点
- `JP_all.txt` - 日本的所有协议节点
- `all_SS.txt` - 所有国家的Shadowsocks节点

**国家代码映射：**

| 国家 | 代码 | 国家 | 代码 |
|------|------|------|------|
| 美国 | US | 加拿大 | CA |
| 英国 | GB | 德国 | DE |
| 荷兰 | NL | 日本 | JP |
| 新加坡 | SG | 香港 | HK |
| 俄罗斯 | RU | 全部 | all |

### H. 错误代码速查表

| 错误信息 | 原因 | 解决方案 |
|----------|------|----------|
| `未选择文件` | 导入时未选择文件 | 点击「选择文件」按钮 |
| `请上传.txt文件` | 文件格式不对 | 确保文件扩展名为 `.txt` |
| `请选择协议` | 未勾选任何协议 | 至少勾选一个协议 |
| `爬取中` | 已有任务在运行 | 等待当前任务完成 |
| `获取失败` | 数据源不可达 | 检查网络，更换数据源 |
| `HTTP 403` | 被目标网站拒绝 | 更换User-Agent，降低请求频率 |
| `HTTP 404` | 数据源链接失效 | 更新 config.json 中的链接 |
| `HTTP 429` | 请求过于频繁 | 减少线程数，增加超时时间 |
| `连接超时` | 网络慢或目标不可达 | 增加超时时间 |
| `内存不足` | 节点数量过多 | 设置最大节点数限制 |
